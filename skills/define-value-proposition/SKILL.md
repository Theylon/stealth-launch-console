# Define Value Proposition

## Overview

Translate customer progress into a value proposition that is specific, testable, and portable into messaging. Use this after positioning when the user needs the crisp promise underneath the copy.

Read [references/frameworks.md](references/frameworks.md) when you need the JTBD and pain-gain lenses.

## Workflow

1. Start with the customer's job, trigger, and desired progress.
2. Map the most relevant pains, frictions, risks, and desired gains for the chosen ICP.
3. Connect the product's differentiated strengths to those outcomes.
4. Draft two or three value proposition options, then recommend one.
5. Call out the proof required to make the claim credible.

## Output Contract

Always produce these sections:

1. `Customer Progress Model`
   - Job, triggers, pains, desired gains
2. `Value Proposition Options`
   - Two or three candidate formulations
3. `Recommended Value Proposition`
   - The strongest single statement
4. `Proof Requirements`
   - What evidence is needed to support the promise
5. `Rationale`
6. `Missing Assumptions`
7. `Suggested Next Skill`
   - Usually `$build-messaging-architecture`

## Quality Bar

- Write value in buyer terms, not in product-team abstractions.
- Tie the promise to one clear job instead of stacking every benefit.
- Avoid unsupported superlatives.
- If the product can only prove a narrower claim, keep the promise narrow.

## Skill Chain

Default handoff: `$build-messaging-architecture`
